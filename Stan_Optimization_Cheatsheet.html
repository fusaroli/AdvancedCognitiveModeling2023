Stan Optimization Cheatsheet

A reference guide to the Stan programming techniques introduced progressively across the book. Each tip includes the chapter and model where it first appears, together with a minimal code example.

1. Block Placement — Run It Once, Not Thousands of Times

Chapter 4 (Memory Model 1, ch06_memory_glm.stan)
Rule 1 callout — “Stan Efficiency Rule 1: Block Placement”

HMC runs thousands of leapfrog steps during warm-up. Every line of Stan code in transformed parameters or model executes at each step; every line in transformed data executes exactly once.

Block Runs Put here
transformed data Once, before sampling Functions of data only — precomputed distances, log-transforms of fixed inputs, cumulative sums
transformed parameters Every leapfrog step (saved) Deterministic functions of parameters whose gradients must propagate — NCP reparameterisations, recursive state, predicted values used in the likelihood
model (local vars) Every leapfrog step (not saved) Intermediate quantities that do not need to be returned — scratch arrays for sequential loops
generated quantities Once per saved draw Posterior predictive samples, log_lik, prior draws — anything expensive that is not needed for sampling

Rule of thumb: If a quantity depends only on data (not on parameters), it belongs in transformed data. A model with 500 trials, 4 chains, and 6 leapfrog steps per draw makes ~24 000 leapfrog evaluations during warm-up. Moving 500 memory values from transformed parameters to transformed data saves ~12 million floating-point operations.

transformed data {
  // computed once — running average of opponent choices
  vector[N] memory;
  memory[1] = 0.5;
  for (t in 2:N)
    memory[t] = (1.0 / t) * sum(other[1:(t-1)]);
}

2. Priors as Data — Avoid Recompilation

Chapter 5 (ch07_SimpleBernoulli_GQ.stan and later multilevel models)
Design Pattern callout — “Stan Design Pattern: Priors as Data”

Pass prior hyperparameters through the data block instead of hard-coding them. Stan recompiles the model whenever the code changes; it never recompiles for new data. This lets you explore prior sensitivity (and run SBC with alternative priors) without recompiling.

data {
  int<lower=0> N;
  array[N] int y;
  // prior hyperparameters passed in as data
  real prior_mu_mean;
  real<lower=0> prior_mu_sigma;
}
model {
  mu ~ normal(prior_mu_mean, prior_mu_sigma);
  y ~ bernoulli_logit(mu);
}
data <- list(N = N, y = y,
             prior_mu_mean = 0, prior_mu_sigma = 1.5)

3. Conditional Diagnostics — The run_diagnostics Flag

Chapter 5 (memory model SBC backend, ch07_Memory_GQ.stan)
Rule 2 callout — “Stan Efficiency Rule 2: Conditional Diagnostics”

Computing log_lik (N values per draw) and y_rep in generated quantities can double runtime. Gate them behind an integer flag so SBC backends can turn them off.

data {
  int<lower=0, upper=1> run_diagnostics;
}
generated quantities {
  vector[N] log_lik = rep_vector(0.0, N);
  array[N] int y_rep;
  if (run_diagnostics) {
    for (i in 1:N) {
      log_lik[i] = bernoulli_logit_lpmf(y[i] | mu);
      y_rep[i]   = bernoulli_logit_rng(mu);
    }
  }
}
# full fit: diagnostics on
data <- list(..., run_diagnostics = 1L)

# SBC: diagnostics off — much faster
sbc_data <- modifyList(data, list(run_diagnostics = 0L))

Cost difference: For 1000 trials, 4 chains, 1000 sampling draws, run_diagnostics = 1 generates and saves 4 million log_lik values. Turning it off during SBC (which only needs the parameters) cuts generated quantities time by ~50%.

4. Non-Centered Parameterization (NCP)

Chapter 6 (multilevel biased agent, ch07_multilevel_biased_ncp.stan)
See also: Chapter 11 (multilevel decay GCM, gcm_ml_decay_stan)

Centered parameterizations create funnel geometries when group-level variance (sigma) is small: the individual-level parameters collapse toward the mean, forming a narrow neck that HMC cannot traverse efficiently (→ many divergences, low E-BFMI, max_treedepth warnings).

Centered (problematic when sigma is small):

parameters {
  real mu;
  real<lower=0> sigma;
  vector[N_subj] theta;     // drawn directly from the population
}
model {
  theta ~ normal(mu, sigma);
}

Non-Centered (always safe — Gaussian geometry throughout):

parameters {
  real mu;
  real<lower=0> sigma;
  vector[N_subj] z;         // standard normal offset
}
transformed parameters {
  vector[N_subj] theta = mu + sigma * z;  // NCP reparameterisation
}
model {
  z ~ std_normal();
}

For simplex parameters (attention weights, mixing proportions), avoid the Dirichlet + kappa parameterization. For a 2-element simplex, use a logit-normal NCP instead:

parameters {
  real pop_logit_w_mean;          // population mean in logit space
  real<lower=0> pop_logit_w_sd;   // population SD in logit space
  vector[N_subj] z_w;             // per-subject NCP offset
}
transformed parameters {
  for (j in 1:N_subj) {
    real logit_w = pop_logit_w_mean + pop_logit_w_sd * z_w[j];
    w[j][1] = inv_logit(logit_w);
    w[j][2] = 1.0 - w[j][1];
  }
}

The centered Dirichlet + exponential(0.1) prior for kappa caused 4000 max_treedepth warnings and near-infinite sampling time in the multilevel decay GCM (Chapter 11). Switching to logit-normal NCP resolved the geometry entirely.

5. Subject Indexing — subj_id vs. Contiguous Blocks

Chapter 6 (multilevel models, ch07_multilevel_memory_ncp.stan)
Rule 3 callout — “Stan Efficiency Rule 3: Subject Indexing”

Two strategies for storing multi-subject trial data in Stan:

Strategy A — subj_id lookup (simple, sequential models only):

data {
  array[N_total] int subj_id;   // which subject each row belongs to
}
model {
  for (i in 1:N_total)
    target += bernoulli_lpmf(y[i] | f(theta[subj_id[i]], x[i]));
}

Strategy B — Contiguous block indexing (required for sequential models and reduce_sum):

data {
  array[N_subj] int subj_start;  // first row index for subject j
  array[N_subj] int subj_end;    // last  row index for subject j
}
model {
  for (j in 1:N_subj) {
    // subject j's trials are rows subj_start[j] .. subj_end[j]
    for (t in subj_start[j]:subj_end[j])
      target += bernoulli_lpmf(y[t] | f(theta[j], x[t]));
  }
}

Strategy B is required whenever: - Trial \(t\) depends on the state accumulated from trials \(1, \ldots, t-1\) (memory, belief updating, RL) - You want to use reduce_sum for within-chain parallelism (see Tip 9)

Build the index in R:

subj_start <- tapply(seq_along(subj_id), subj_id, min)
subj_end   <- tapply(seq_along(subj_id), subj_id, max)

6. Pathfinder as Warm-Start Initialisation

Chapter 4 (logit-bias model benchmark, section “Benchmarking NUTS, Pathfinder, and Pathfinder-initialized NUTS”)
Chapter 6 (NCP multilevel model timing comparison)
Chapter 11 (decay GCM 3-way timing benchmark)

Stan’s default initialisation draws chain start points from Uniform(−2, 2) in unconstrained space — far from the posterior on hierarchical models. Pathfinder runs a fast variational approximation (L-BFGS path + Gaussian fits) and returns approximate posterior draws. Passing these as init cuts the warm-up phase by 30–60% on models with challenging geometry.

# Step 1: run Pathfinder
pf <- mod$pathfinder(data = stan_data, num_paths = 4, draws = 1000,
                     seed = 123)

# Step 2: use Pathfinder draws as starting points for NUTS
init_pf <- lapply(1:4, function(i) as.list(pf$draws()[i, , ]))

fit <- mod$sample(data = stan_data, init = init_pf,
                  chains = 4, iter_warmup = 500, iter_sampling = 1000)

When it helps most: hierarchical/multilevel models, NCP models with many group-level parameters, models with correlated population parameters. Less benefit for simple, well-conditioned models where the default initialisation already lands near the posterior.

Note: Pathfinder itself is not a gold-standard posterior. Use it only for initialisation; always run full MCMC and inspect diagnostics.

7. Marginalizing Discrete Parameters with log_mix

Chapter 8 (mixture models, single_mixture_stan and multilevel_mixture_stan)
Technique callout — “Stan Technique: Marginalizing Discrete Parameters with log_mix”

HMC requires continuous, differentiable parameters. Discrete latent variables (component indicators, regime switches) must be analytically marginalized out of the likelihood. Stan’s log_mix function does this for 2-component mixtures in one line.

// Component 1 log-likelihood
real lp1 = bernoulli_logit_lpmf(y | alpha);
// Component 2 log-likelihood
real lp2 = bernoulli_logit_lpmf(y | nu);
// Marginalise: log(pi * exp(lp1) + (1-pi) * exp(lp2))
target += log_mix(pi, lp1, lp2);

For \(K > 2\) components, use log_sum_exp manually:

vector[K] lps;
for (k in 1:K)
  lps[k] = log(pi[k]) + likelihood_lpmf(y | theta[k]);
target += log_sum_exp(lps);

Never sample a discrete indicator z with z ~ categorical(pi) and then condition on it — HMC cannot compute gradients with respect to discrete parameters.

8. Sequential State Unrolling in transformed parameters

Chapter 9 (Theory of Mind, WeightedAgentRaw_stan and multilevel SBA/PBA/WBA models)
Technique callout — “Stan Technique: Sequential State Unrolling in transformed parameters”

When a model’s likelihood depends on a latent state that evolves sequentially (belief, running weight, RL value), unroll the state recursion in transformed parameters so the gradient can flow back through the entire trajectory.

transformed parameters {
  vector[N] predicted_choice;
  {
    // local workspace — allocated and freed each leapfrog step
    real belief = 0.5;
    for (t in 1:N) {
      predicted_choice[t] = inv_logit(weight * (belief - 0.5));
      // update belief based on observed feedback
      belief += alpha * (feedback[t] - belief);
    }
  }
}
model {
  target += bernoulli_lpmf(y | predicted_choice);
}

Why transformed parameters and not model? - State that flows into the likelihood needs its gradient tracked by autodiff. - The {} local scope inside transformed parameters keeps scratch variables off the saved-draw tape, reducing memory allocation. - Exception: if the sequential loop is independent across subjects and you want reduce_sum parallelism, move the per-subject loop into the partial_sum_lp function and keep only the output vector in transformed parameters.

9. Intra-Chain Parallelism with reduce_sum

Chapter 11 (multilevel decay GCM, gcm_ml_decay_parallel_stan)
Rule 4 callout — “Stan Efficiency Rule 4: Intra-Chain Parallelism via reduce_sum”

reduce_sum exploits conditional independence across subjects: given the population parameters, subject \(j\)’s trial loop shares no mutable state with subject \(k\)’s loop. The subject loop can therefore be distributed across CPU threads within a single chain.

Step 1 — Write a partial_sum_lp function:

functions {
  real partial_sum_lp(
    array[] int subj_slice, int start, int end,
    // all parameters and data the function needs
    vector log_c, array[] int subj_start_g, array[] int subj_end_g,
    array[] int y, array[,] real x
  ) {
    real lp = 0.0;
    for (idx in 1:size(subj_slice)) {
      int j = subj_slice[idx];
      for (t in subj_start_g[j]:subj_end_g[j])
        lp += bernoulli_lpmf(y[t] | f(exp(log_c[j]), x[t]));
    }
    return lp;
  }
}

Step 2 — Replace the subject loop with reduce_sum:

model {
  // subject_ids = {1, 2, ..., N_subjects}
  target += reduce_sum(partial_sum_lp, subject_ids, 1,
                       log_c, subj_start, subj_end, y, x);
}

Step 3 — Compile and sample with threads:

mod_par <- cmdstan_model(stan_file,
                         cpp_options = list(stan_threads = TRUE))

n_threads <- max(1L, parallel::detectCores() %/% 4L)

fit_par <- mod_par$sample(
  data = stan_data,
  chains = 4, parallel_chains = 4,
  threads_per_chain = n_threads,
  iter_warmup = 500, iter_sampling = 1000
)

When is reduce_sum beneficial?

Condition Effect
Many subjects (≥ 20) with expensive per-subject loops Large speedup (near-linear up to N_subjects)
Few subjects or cheap per-subject loops Overhead dominates; may be slower
Within-subject sequential dependency Still valid — parallelises across subjects, not within
subj_start / subj_end contiguous indexing Required (see Tip 5)

Note: Within-subject trial loops (as in the single-subject GCM, where trial \(t\) depends on the memory accumulated from trials \(1, \ldots, t-1\)) cannot be parallelised with reduce_sum. The parallelism is always across the independent subject dimension.

10. Combining Techniques — Compound Speedups

Chapter 11 (3-way timing benchmark ch12_timing_benchmark)

The techniques above stack multiplicatively on hierarchical models:

Strategy Typical saving When to apply
NCP instead of CP 5–100× fewer divergences Any multilevel model; always prefer NCP
transformed data precompute 10–100× fewer flops Static quantities derived from data only
run_diagnostics = 0 in SBC ~2× faster generated quantities All SBC backends
Pathfinder warm-start 20–50% less wall time Hierarchical models; models with correlated posteriors
reduce_sum (10 subjects, 4-core laptop) 2–3× within-chain ≥ 20 subjects, expensive loops, stan_threads = TRUE
Pathfinder + reduce_sum together 3–5× vs. serial baseline Large multilevel models on multi-core hardware

Benchmark template (used in Chapters 6 and 11):

time_serial_nopf <- system.time(fit_serial <- mod$sample(...))[["elapsed"]]
pf              <- mod$pathfinder(...)
time_serial_pf  <- system.time(fit_pf <- mod$sample(init = pf_inits, ...))[["elapsed"]]
time_parallel   <- system.time(fit_par <- mod_par$sample(
                     init = pf_inits, threads_per_chain = n_threads, ...))[["elapsed"]]

tibble(
  Strategy   = c("Serial (no PF)", "Serial + Pathfinder", "Parallel + Pathfinder"),
  Time_sec   = c(time_serial_nopf, time_serial_pf, time_parallel),
  Speedup    = Time_sec[1] / Time_sec
)
